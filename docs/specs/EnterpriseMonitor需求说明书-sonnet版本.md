## 1. 产品定位与核心理念

### 1.1 从"运营看板"到"企业智能仪表盘"

传统运营看板是**后视镜**——对已发生的内部数据做客观统计与展现，按月按季更新，信息孤立，无法指导决策。

**EnterpriseMonitor** 是**前挡风玻璃**——实时感知企业内部运营脉搏与外部经营环境，由 AI 持续分析、主动预警、输出结论，为管理层提供随时可用的决策依据。

### 1.2 设计原则

| 原则 | 传统看板 | EnterpriseMonitor |
|------|---------|-------------------|
| 信息来源 | 仅内部系统 | 内部 + 外部，各占一半 |
| 更新频率 | 月/季度 | 实时（秒级到分钟级）|
| 信息密度 | 稀疏，大量留白 | 高密度，参考 WorldMonitor 风格 |
| AI 角色 | 无 | 全程参与：分析、预警、结论输出 |
| 用户姿态 | 主动查询 | 主动推送 + 被动查询 |

### 1.3 类比关系

```
WorldMonitor          →    EnterpriseMonitor
─────────────────────────────────────────────
地缘政治情报           →    企业经营情报
国家不稳定指数(CII)    →    企业健康指数(EHI)
军事活动追踪           →    销售/交付进展追踪
基础设施级联分析       →    供应商/客户依赖分析
DEFCON 警报           →    企业风险预警等级
AI 战略态势面板        →    AI 经营态势面板
预测市场赔率           →    业务预测概率
```

## 2. 企业全领域元数据模型

> **这是所有功能的地基。** EnterpriseMonitor 展示的每一条信息，背后都有一个结构化的企业领域对象支撑。没有这套元数据模型，"高密度"只是堆砌数字，"AI分析"只是胡说八道。

### 2.1 核心领域对象（Domain Objects）

```
企业（Enterprise）
├── 组织（Organization）
│   ├── 法人主体（LegalEntity）：公司/分公司/子公司
│   ├── 部门（Department）：组织树节点
│   ├── 岗位（Position）：职级体系
│   └── 人员（Employee）：雇员档案
│
├── 客户（Customer）
│   ├── 客户主档（CustomerProfile）
│   ├── 联系人（Contact）
│   ├── 商机（Opportunity）
│   └── 合同（Contract）
│
├── 产品（Product）
│   ├── 产品线（ProductLine）
│   ├── 产品版本（ProductVersion）
│   ├── SKU（StockKeepingUnit）
│   └── 定价（Pricing）
│
├── 项目（Project）
│   ├── 项目档案（ProjectProfile）：交付类/研发类/内部类
│   ├── 里程碑（Milestone）
│   ├── 任务（Task）
│   └── 资源分配（ResourceAllocation）
│
├── 财务（Finance）
│   ├── 收入（Revenue）：合同额/回款/确认收入
│   ├── 成本（Cost）：人力/外包/直接成本
│   ├── 费用（Expense）：销售费用/管理费用
│   └── 现金流（CashFlow）
│
├── 供应商（Supplier）
│   ├── 供应商主档（SupplierProfile）
│   ├── 采购合同（PurchaseContract）
│   └── 付款（Payment）
│
├── 人力资源（HumanResource）
│   ├── 招聘（Recruitment）
│   ├── 考勤（Attendance）
│   ├── 绩效（Performance）
│   └── 薪酬（Compensation）
│
└── 外部环境（ExternalEnvironment）
    ├── 政策法规（Policy）
    ├── 行业动态（Industry）
    ├── 竞争对手（Competitor）
    ├── 宏观经济（Macroeconomics）
    └── 市场情报（MarketIntelligence）
```

### 2.2 领域对象的核心属性（通用字段）

每个领域对象都携带以下通用元数据，使 AI 得以跨领域关联分析：

| 字段 | 类型 | 说明 |
|------|------|------|
| `entity_id` | UUID | 全局唯一标识 |
| `entity_type` | Enum | 所属领域类型 |
| `created_at` | Timestamp | 创建时间 |
| `updated_at` | Timestamp | 最后更新时间（实时性的锚点）|
| `status` | Enum | 当前状态（实体生命周期）|
| `owner_id` | UUID | 责任人（员工 ID）|
| `org_path` | String | 组织路径（支持权限隔离）|
| `tags` | Array | 多维标签（用于 AI 分类）|
| `health_score` | Float[0-100] | AI 计算的健康度评分 |
| `risk_flags` | Array | AI 标注的风险信号 |
| `related_entities` | Array | 关联实体列表（支持级联分析）|
| `external_signals` | Array | 关联外部信号（内外打通的关键）|

### 2.3 关键关系链

```
客户 ──── 签署 ────► 合同 ──── 驱动 ────► 项目
  │                   │                    │
  │                 回款状态             交付进度
  │                   │                    │
  └──── 影响 ────► 收入预测 ◄──── 影响 ───┘

员工 ──── 分配到 ──► 项目 ──── 消耗 ────► 人力成本
  │                                         │
考勤/绩效                              毛利率计算
  │                                         │
  └──── 影响 ──────────────────────────────┘

外部政策/行业动态 ──── AI关联 ────► 相关客户/项目/产品线
```

## 3. 维度一：内外部信息打通

### 3.1 设计原则

传统看板 100% 内部数据。EnterpriseMonitor 目标：**内部 50% + 外部 50%**，且两者必须关联展示，而非简单并列。

外部信息的价值不在于"有"，而在于**与内部业务的关联**：  
- 某客户所在行业监管政策收紧 → 该客户合同续约风险上升  
- 某地区发生自然灾害 → 在该地有项目的交付风险预警  
- 央行降息 → 客户资金成本下降 → 有利于大额合同决策  

### 3.2 外部信息类别体系

#### 3.2.1 政治与监管层

| 信息类型 | 数据来源 | 关联内部对象 | 更新频率 |
|---------|---------|------------|---------|
| 国家/地方政策发布 | 政府官网 RSS、政策数据库 | 客户、产品线、所在地项目 | 实时 |
| 行业监管动态 | 行业协会、监管机构公告 | 相关行业客户、合规部门 | 实时 |
| 政府采购政策 | 中国政府采购网 | 政府类客户、招投标项目 | 实时 |
| 法规修订 | 法律法规数据库 | 合同条款、合规风险 | 实时 |
| 领导人动态/重要讲话 | 新华社、人民日报 | 战略方向、政策预判 | 实时 |

#### 3.2.2 经济与市场层

| 信息类型 | 数据来源 | 关联内部对象 | 更新频率 |
|---------|---------|------------|---------|
| 宏观经济指标（GDP/CPI/PMI）| 国家统计局、Wind | 营收预测、客户预算 | 月度+实时预览 |
| 利率/汇率变动 | 央行、外汇局 | 财务成本、外币合同 | 实时 |
| 行业景气指数 | 行业协会、研究机构 | 各业务线增长预测 | 月度 |
| 竞争对手动态 | 公司公告、新闻、招聘信息 | 产品竞争力、定价策略 | 实时 |
| 融资/并购信息 | 企查查、天眼查、新闻 | 客户财务健康、潜在风险 | 实时 |

#### 3.2.3 行业与技术层

| 信息类型 | 数据来源 | 关联内部对象 | 更新频率 |
|---------|---------|------------|---------|
| 行业招投标公告 | 政采云、各省采购平台 | 销售机会、商机预测 | 实时 |
| 技术趋势与标准 | 工信部、IEEE、arxiv | 产品研发方向 | 实时 |
| 客户企业新闻 | 新闻聚合、客户官网 | 客户健康度、决策人变动 | 实时 |
| 上市客户财报 | 证交所、Wind | 客户预算预测 | 季度+实时 |
| 人才市场动态 | 招聘网站、薪酬报告 | HR 成本基准、人才争夺 | 月度 |

#### 3.2.4 社会与环境层

| 信息类型 | 数据来源 | 关联内部对象 | 更新频率 |
|---------|---------|------------|---------|
| 重大自然灾害 | USGS、应急管理部 | 项目所在地风险 | 实时 |
| 疫情/公共卫生事件 | 卫健委、WHO | 人员出行、现场项目 | 实时 |
| 重大社会事件 | 新闻聚合 | 相关行业客户情绪 | 实时 |
| 互联网/网络中断 | Cloudflare Radar 等 | SaaS 服务可用性 | 实时 |

### 3.3 内外部关联规则引擎

系统维护一张**关联规则表**，当外部事件发生时，自动在内部对象上标注影响：

```
外部事件：[政策] 某行业监管收紧
  → 规则匹配：客户.行业 = 该行业
  → 影响评估：合同续约风险 +30分
  → 界面标注：该客户卡片显示 ⚠ 政策风险信号
  → AI分析：自动生成影响摘要推送给客户负责人

外部事件：[市场] 某客户被曝资金困难
  → 规则匹配：客户.entity_id 匹配
  → 影响评估：回款风险 +50分，合同风险 CRITICAL
  → 界面标注：该客户在所有面板中变红，发出 ALERT
  → AI分析：列出该客户名下所有未回款合同金额
```

## 4. 维度二：信息实时性

### 4.1 实时性分级体系

不同信息的实时性要求不同，按价值和技术可行性分为四级：

| 级别 | 刷新周期 | 适用信息类型 | 实现方式 |
|------|---------|------------|---------|
| **L1 秒级** | ≤10秒 | 警报、严重异常、系统故障 | WebSocket 推送 |
| **L2 分钟级** | 1-5分钟 | 外部新闻、市场数据、工作流状态 | SSE / 轮询 |
| **L3 小时级** | 15-60分钟 | 项目进度、销售漏斗、财务日报 | 定时任务 |
| **L4 日级** | 每日 | 人效数据、月度预测更新、KPI | 夜间批处理 |

### 4.2 内部实时数据接入

| 数据类型 | 原始系统 | 接入方式 | 延迟目标 |
|---------|---------|---------|---------|
| 工作流审批状态 | 项目协管、OA | Webhook | L1 |
| 合同签署状态 | 合同系统 | Webhook | L1 |
| 回款到账 | 财务系统 | Webhook + 银企直连 | L1 |
| 任务状态变更 | 协作平台 | Webhook | L2 |
| CRM 商机更新 | CRM 系统 | CDC（Change Data Capture）| L2 |
| 考勤异常 | 假勤系统 | 推送 | L2 |
| 项目里程碑 | 项目管理 | CDC | L2 |
| 日报/周报数据 | 工作量填报 | L3 定时聚合 | L3 |
| 薪酬/绩效数据 | HR 系统 | L4 批处理 | L4 |

### 4.3 外部实时数据接入

| 数据类型 | 来源 | 接入方式 | 延迟目标 |
|---------|------|---------|---------|
| 政策法规发布 | 政府官网 RSS | RSS 轮询 | L2 |
| 招投标公告 | 政采云 API | API 轮询 | L2 |
| 重大新闻 | 新闻聚合 RSS | RSS 轮询 | L2 |
| 上市客户股价 | 证券数据接口 | WebSocket | L2 |
| 宏观数据 | Wind/Choice API | 定时拉取 | L3 |
| 竞争对手动态 | 新闻+企查查 | 定时爬取 | L3 |

### 4.4 时间轴回放

参考 WorldMonitor 的时间轴筛选器，EnterpriseMonitor 支持：

```
[1h] [6h] [24h] [7d] [30d] [本季] [自定义]
```

任何面板均可回到历史某时刻的状态，与今日对比（快照 Diff 功能）。

## 5. 维度三：信息密度与丰富度

### 5.1 设计目标

对标 WorldMonitor 的界面密度，传统看板信息密度增加 **3-5倍**。核心方法：

1. **压缩冗余留白**：卡片间距最小化，数字大、标签小
2. **多层信息叠加**：每个数字旁附带趋势、环比、AI评级
3. **颜色即信息**：红/橙/黄/绿/蓝五色语义系统，无需阅读就能感知状态
4. **图表即数据**：用 Sparkline 迷你图替代纯数字，一行放下趋势
5. **标签即维度**：每条信息附 ALERT / RISK / GROWTH / NORMAL 状态标签

### 5.2 信息密度参考设计

#### 单个客户卡片（传统 vs EnterpriseMonitor）

**传统看板**（一张卡片，显示 2 个字段）：
```
┌─────────────────────┐
│  客户：某某公司       │
│  合同额：500万        │
└─────────────────────┘
```

**EnterpriseMonitor**（同等空间，显示 12+ 个字段）：
```
┌──────────────────────────────────────────────┐
│ 某某公司  [政府-电子政务]  ⚠ POLICY RISK       │
│ ─────────────────────────────────────────── │
```

## 6. 维度四：AI 参与度

### 6.1 AI 角色定义

传统看板中 AI 缺席。EnterpriseMonitor 中 AI 扮演三个角色：

| 角色 | 类比 | 职责 |
|------|------|------|
| **情报分析官** | WM 的 AI 摘要引擎 | 持续消化内外部信息流，生成摘要与洞察 |
| **风险预警员** | WM 的收敛检测 | 识别异常模式，主动发出预警 |
| **决策参谋** | WM 的 AI 预测预报 | 基于数据趋势，给出建议与预测概率 |

### 6.2 AI 分析流水线

```
数据采集层（内部系统 + 外部信源）
         │
         ▼
实体识别与关联（将新闻/政策与内部领域对象挂钩）
         │
         ▼
信号分类与优先级排队（CRITICAL / HIGH / MEDIUM / LOW）
         │
         ▼
AI 摘要生成（分板块：客户/项目/财务/外部）
         │
         ▼
结论输出（不只是"有什么"，而是"说明什么"、"要怎么做"）
         │
         ▼
推送与展示（面板 + 消息通知 + 晨报/周报）
```

### 6.3 企业健康指数（EHI）—— 对标 WM 的 CII

EHI（Enterprise Health Index）是 EnterpriseMonitor 的核心 AI 输出，对每个业务实体给出 0-100 的综合评分。

#### 6.3.1 客户健康指数（Customer Health Index, CHI）

| 分项 | 权重 | 计算依据 |
|------|------|---------|
| R - 回款健康 | 30% | 回款率、账龄分布、逾期金额占比 |
| E - 参与度 | 20% | 近期沟通频率、需求提交、满意度 |
| D - 交付满意 | 25% | 项目里程碑达成率、验收通过率 |
| O - 外部信号 | 25% | 客户企业新闻、财务状况、政策风险 |

```
CHI = R×0.3 + E×0.2 + D×0.25 + O×0.25
```

颜色编码：
- 80-100：绿色（健康）
- 60-79：蓝色（正常）
- 40-59：黄色（关注）
- 20-39：橙色（风险）
- 0-19：红色（危险）

#### 6.3.2 项目健康指数（Project Health Index, PHI）

| 分项 | 权重 | 计算依据 |
|------|------|---------|
| S - 进度 | 35% | 里程碑达成率，计划偏差天数 |
| Q - 质量 | 25% | Bug率、验收一次通过率、返工次数 |
| C - 成本 | 25% | 实际工时 vs 预算工时，毛利率 |
| T - 团队 | 15% | 核心人员稳定性，工时饱和度 |

#### 6.3.3 部门健康指数（Department Health Index, DHI）

| 分项 | 权重 | 计算依据 |
|------|------|---------|
| P - 绩效达成 | 40% | OKR/KPI 达成率 |
| H - 人员健康 | 30% | 离职率、工时异常、满意度 |
| W - 工作流效率 | 30% | 待办处理时效、审批周期 |

#### 6.3.4 企业整体健康指数（EHI）

```
EHI = CHI均值×0.35 + PHI均值×0.30 + DHI均值×0.20 + 外部环境评分×0.15
```

EHI 实时计算，精确到分钟，顶部导航栏常驻显示当前值与趋势箭头。

### 6.4 AI 信号检测（12 类信号）

**业务信号**

| 信号 | 触发条件 | 含义 |
|------|---------|------|
| ◉ 收入收敛 | 3+ 大单同期进入回款关键节点 | 本月/季度收入有望超预期 |
| ⚠ 回款预警 | 合同回款超期 30 天且无跟进记录 | 坏账风险上升 |

| ❄ 商机停滞 | 商机 7 天无推进且处于关键阶段 | 需要销售主管介入 |

**交付信号**

|------|---------|------|
| 🚨 交付风险 | 项目里程碑偏差 > 10 天 | 需要资源调配或客户沟通 |
| 📉 毛利侵蚀 | 项目实际工时超预算 20%+ | 项目盈利能力受损 |
| 👤 关键人风险 | 项目核心成员离职/长假 | 交付连续性风险 |

**人力信号**

| 信号 | 触发条件 | 含义 |

### 6.6 AI 晨报与闪报

**晨报（每日 08:30 推送）**：
- 昨日经营摘要（收入/回款/新签/完成里程碑）
- 今日重点待办（按角色个性化：CEO看全局，PM看项目，销售看商机）
- 外部情报摘要（政策/市场/竞对）
- AI 重点关注（3条最高优先级的 AI 洞察）

**闪报（事件触发，实时推送）**：
- CRITICAL 级别信号立即推送
- HIGH 级别信号聚合后每小时推送
- 推送渠道：企业 IM（品高聆客）、邮件、短信（仅 CRITICAL）

## 10. 权限与安全架构

### 10.1 角色体系

| 角色 | 可见数据范围 | 典型用户 |
|------|-----------|---------|
| 决策层（C-Suite）| 全公司所有数据 + 全部外部情报 | CEO、CFO、COO |
| 事业部负责人 | 本事业部全部数据 + 相关外部情报 | VP、总监 |
| 部门经理 | 本部门数据 + 负责项目/客户 | 部门总监、PM |
| 一线员工 | 个人负责项目/客户 | 开发、销售、顾问 |
| 数据分析师 | 数据只读（脱敏）| BI 分析师 |

### 10.2 数据分级

| 级别 | 数据类型 | 说明 |
|------|---------|------|
| L1 公开 | 外部情报、行业新闻 | 全员可见 |
| L2 内部 | 项目状态、任务、动态 | 按项目成员可见 |
| L3 敏感 | 合同金额、客户联系人 | 按负责人可见 |
| L4 机密 | 薪酬、绩效、利润率 | 仅管理层 |
| L5 最高机密 | 战略规划、并购信息 | 仅 C-Suite |

### 10.3 AI 安全策略

- AI 模型本地部署优先（私有化部署），不将内部数据发送至公有云 LLM
- AI 生成内容需标注数据来源，确保可追溯
- 对话式查询记录完整审计日志
- 数据脱敏规则：下级角色通过 AI 查询时，不返回超出其权限的数据

- 供应商依赖分析上线

- 面向 C-Suite 的个性化视图深度优化
